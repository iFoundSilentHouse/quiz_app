🏁 Шаг 6 — Прохождение теста и проверка результатов
1. Страница прохождения (/quiz/[id])

Реализована логика «клиентского прохождения». Основные фишки:

    Идентификация: Добавлено поле studentName. Без имени кнопка завершения заблокирована или вызывает предупреждение.

    Сбор ответов: Использовано состояние Record<number, string>, где ключ — ID вопроса, а значение — введенный текст. Это позволило избежать путаницы при перемешивании вопросов.

    Интеграция с API: Данные отправляются на эндпоинт POST /quizzes/:id/attempts.

2. Логика проверки на бэкенде (AttemptService)

Бэкенд теперь не просто сохраняет текст, а проводит мгновенный аудит:

    Нормализация: Сравнение ответов идет с использованием .trim().toLowerCase(), чтобы случайные пробелы или регистр не влияли на оценку.

    Расчет Score: Сервер инкрементирует счетчик только при полном совпадении строк.

    Генерация Diff: Для каждого ответа формируется массив DiffCharDto[].

3. Компонент DiffHighlight

Критически важный UI-элемент для обратной связи. Он визуализирует массив объектов, где каждый символ имеет свой статус:

    correct: Зеленый цвет (буква совпала).

    incorrect: Красный цвет + подчеркивание (ученик ошибся в букве).

    extra: Красный фон (ученик ввел лишние символы).

    missing: Серый цвет (символы, которые ученик забыл ввести).

4. Страница результата (/quiz/[id]/result)

Информационный дашборд для ученика после завершения:

    Итоговая карточка: Крупный индикатор баллов (например, 3 / 5).

    Детальный разбор: Список всех карточек, где сопоставляются:

        Оригинальное изображение.

        Визуализированный «Ваш ответ» (через DiffHighlight).

        Эталонный правильный ответ (снизу для сравнения).

5. Инфраструктурные изменения (из diff)

    CORS Policy: В main.ts добавлены явные методы PUT, PATCH, DELETE. Это было необходимо, так как прохождение теста инициирует сложные кросс-доменные запросы.

    Зависимости: Добавлена библиотека heroicons для визуальных индикаторов (иконки копирования, успеха и ошибок).

    Environment: Внедрена переменная FRONTEND_PORT, чтобы бэкенд точно знал, какому адресу разрешать доступ к данным.
